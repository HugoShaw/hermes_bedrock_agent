## Scope Definition

### v0.2.0 Scope

Single-version dual-path retrieval debugging for one selected `project_id` / `experiment_project_id` at a time.

```text
1. Improve visibility into LanceDB vector retrieval.
2. Improve visibility into Neptune graph retrieval.
3. Show graph-guided retrieval status (strong/weak/none).
4. Show keyword boost details.
5. Show graph edge confidence/status information.
6. Warn about low-confidence or candidate graph relationships.
7. Warn about project_id / experiment_project_id isolation issues.
8. Show timing breakdown for vector, graph, context assembly, answer generation.
9. Keep normal QA terminal usage simple and unchanged by default.
10. Support one selected project_id / experiment_project_id at a time.
```

## 1. Current QA Terminal Architecture

### Overview

The QA terminal (`src/hermes_bedrock_agent/qa/terminal.py`, 594 lines) is an interactive REPL providing three modes:

| Mode | Trigger | Description |
|------|---------|-------------|
| `retrieve` | `/mode retrieve` | Vector-only retrieval, shows chunks |
| `graph` | `/mode graph` | Vector + Neptune graph context |
| `answer` (default) | `/mode answer` | Full dual-path retrieval + LLM answer generation |

### Entry Points

1. **CLI one-shot:** `dualrag qa "question" --project-id <id> --mode answer`
2. **CLI interactive:** `dualrag qa --project-id <id>` → enters REPL loop

### Current CLI Parameters

From `cli.py` lines 328-337:
- `--project-id` — scopes retrieval
- `--collection` — overrides LanceDB table (for experiment data)
- `--mode` — retrieve | answer | graph
- `--top-k` — number of chunks (default: 5)
- `--no-graph` — skip Neptune entirely
- `--log-level` — logging verbosity (default: WARNING)

### Interactive Commands

| Command | Effect |
|---------|--------|
| `/help` | Show available commands |
| `/mode <m>` | Switch retrieve/answer/graph |
| `/verbose` | Toggle verbose chunk display |
| `/top <n>` | Change top-k |
| `/project <id>` | Switch project scope |
| `/exit`, `/quit`, `Ctrl+D` | Exit |

## 2. Current LanceDB Vector Retrieval Flow

### Pipeline

```text
query_text
  → Titan Embed v2 (Bedrock, 1024 dims)
  → LanceDB ANN search
  → prefilter: project_id = '<id>' [AND sheet_index IN (...)]
  → top_k results
  → RetrievedChunk objects (score = 1 / (1 + L2_distance))
```

### Key Code: `knowledge_base/vector_store.py`

- `query_vector_store(query_text, cfg, top_k, store_path, collection, project_id, sheet_filter)`
- Prefilter: LanceDB WHERE clause, applied before ANN
- Optional `sheet_filter` from graph guidance

### Key Code: `retrieval/vector_retriever.py`

- `retrieve_chunks(query, top_k, cfg, project_id, collection)` — thin wrapper
- Converts raw dicts → `RetrievedChunk` objects

### Current Limitations

1. No trace output — vector search is opaque; user sees only final chunks
2. No raw distance values exposed
3. No embedding latency visible
4. No indication of which filter was applied or how many candidates were evaluated

## 3. Current Neptune Graph Retrieval Flow

### Pipeline (Two-Layer Architecture)

```text
query_text
  → extract_entity_names(query)  [system keywords, API patterns]
  → _extract_query_keywords(query)  [kanji, katakana, Latin ≥4 chars]
  → Neptune: MATCH (n) WHERE toLower(n.name) CONTAINS toLower($term)
  → expand 1-hop: (n)-[r]-(Sheet) → sheet_indices
  → expand 1-hop: (n)-[r]-(System) → relevant_systems
  → evaluate_hint_quality(hints) → "strong" / "weak" / "none"
  → (if strong) filtered vector search + standard search + merge
  → (if weak)   standard vector search only, graph for context
  → build DualGraphContext:
      → Business: System-System edges, Sheet neighbourhood
      → Implementation: Sheet→Field/API/Rule, Entity-name search
  → keyword_boost(chunks) [+0.02/match, max +0.08]
  → return (chunks, dual_graph, guidance_status)
```

### Key Code: `retrieval/graph_guided_retrieval.py` (716 lines)

- `explore_graph_for_query(query, project_id)` → `GraphGuidanceHints`
- `_build_dual_graph_from_hints(client, hints, query, project_id)` → `DualGraphContext`
- `retrieve_with_graph_guidance(query, top_k, project_id, cfg)` → orchestrator
- `_merge_chunks(guided, standard, guided_boost=0.05)` — deduplicate and boost

### Key Code: `retrieval/graph_retriever.py` (478 lines)

- `fetch_dual_graph_context(chunks, query, project_id)` — chunk-driven fallback
- `_fetch_business_graph(client, chunks, query, project_id)` — 4 strategies
- `_fetch_implementation_graph(client, chunks, query, project_id)` — 4 strategies (A-D)
  - Strategy D: Keyword fallback for experiment graphs

### Graph Hint Quality Evaluation

- **Strong**: ≤10 sheets AND < 50% of total → filter vector search
- **Weak**: >10 sheets OR > 50% of total → graph context only
- **None**: no matching nodes → pure vector search

### Current Limitations

1. Graph guidance decision invisible to user
2. No per-term match results shown
3. Hint quality thresholds not explained
4. No edge confidence/status properties checked
5. Keyword boost amounts invisible
6. Neptune query count and latency hidden

## 4. Current Context Assembly Flow

### Answer Mode Pipeline

```text
_handle_answer (terminal.py ~line 360):
  → retrieve_with_graph_guidance(query, top_k, project_id)
  → load_evidence_images(chunks, project_root)
  → _print_chunks(chunks)
  → _print_dual_graph(dual_graph)
  → _print_evidence_flow(chunks, dual_graph, evidence_images, elapsed)
  → generate_answer(query, chunks, evidence_images, business_graph, implementation_graph)
  → _print_answer(ans_resp)
```

### Context Sent to LLM (answer_generator.py)

```text
System prompt: Japanese enterprise integration specialist (~500 chars)
User message:
  1. "## Retrieved Markdown Chunks" — text with score/metadata
  2. "## Business Semantic Graph" — serialized nodes + edges
  3. "## Implementation Graph" — serialized nodes + edges
  4. "## Visual Evidence" — PDF/PNG images (multimodal)
  5. "Please answer this question: <query>"
```

### Current Limitations

1. Graph context passed unfiltered — no confidence indicator on edges
2. Candidate/POSSIBLY_RELATED edges pass through without warning
3. No timing breakdown visible

## 6. Current Project ID Isolation Mechanism

### Vector Store (LanceDB) — Strong Isolation

- `project_id` column in schema
- Filter: `project_id = '<id>'` as prefilter (before ANN)
- Different project IDs never mix in vector results
- Experiment data: separate collection `dualrag_experiment_eval_chunks`

### Graph Store (Neptune) — Weak Isolation

- `project_id` property on nodes (not always present)
- Filter: `n.project_id = $pid OR n.project_name = $pid`
- Many nodes lack `project_id` (connected via Workbook/Project relationships)
- Neighbor expansion: **inconsistently filtered** — some queries check `m.project_id`, some don't
- Experiment nodes: prefixed IDs like `exp_saimu_bugyo_cloud__chunk_fixed_length__graph_v43::node_name`

### Known Issues

1. Cross-project leakage in graph expansion
2. Missing project_id on shared nodes (System: "SAP", "DataSpider")
3. Experiment graph uses different labels than production
4. No visual indication of isolation status to user

## 7. Current Limitations Summary

| # | Limitation | Impact |
|---|-----------|--------|
| 1 | No retrieval trace visible to user | Cannot debug wrong chunk/node selection |
| 2 | Graph guidance decision opaque | Cannot tell if graph helped or hurt |
| 3 | No confidence/status on graph edges | Low-quality edges treated same as high-quality |
| 4 | No CANDIDATE/POSSIBLY_RELATED filtering | Noisy context may mislead LLM |
| 5 | keyword_boost invisible | Cannot verify if boost improves ranking |
| 6 | Cross-project leakage in graph | Shared nodes pull in unrelated data |
| 7 | No timing breakdown | Cannot identify latency bottleneck |
| 8 | No vector-only toggle at runtime | Cannot isolate vector-path contribution |

## 8. Proposed Target Architecture (v0.2.0)

### Design Principles

1. **Default mode unchanged** — current UX remains clean and simple
2. **Trace is opt-in** — activated by debug flags, not always-on
3. **Graph enhances, never overrides** — confidence gating prevents noise
4. **Strict isolation is opt-in** — warns by default, errors only with `--strict`
5. **Backward compatible** — all existing CLI commands produce identical output
6. **Single-version only** — one project_id per session, no cross-experiment features

### Architecture Diagram

```text
┌────────────────────────────────────────────────────────────────────┐
│ QA Terminal Session (single project_id)                             │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │ Retrieval Orchestrator (enhanced)                           │   │
│  │  - Timing instrumentation (per-stage)                       │   │

│  │  - Project isolation checker                                │   │
│  └─────────┬──────────────────────────────────┬────────────────┘   │
│            │                                  │                    │
│  ┌─────────▼──────────┐      ┌───────────────▼────────────────┐   │
│  │ Vector Path         │      │ Graph Path                      │   │
│  │  - Embed + ANN      │      │  - Term extraction              │   │
```

## 16. Backward Compatibility Risks

| Risk | Mitigation |
|------|-----------|
| New CLI flags could conflict | All flags have unique long names; no short-flag collision |
| Trace collection overhead | Trace dicts allocated only when debug flags active |
| Confidence filtering reduces answer quality | Threshold defaults to 0.0 (no filtering); opt-in only |
| Edge property reading fails on old data | Graceful fallback: missing confidence → treat as 1.0 (pass through) |
| `--vector-only` and `--no-graph` overlap | `--vector-only` is alias; same code path |

### Guaranteed Unchanged

1. `dualrag qa "question" --project-id X` — identical output
2. `dualrag qa` (interactive) — same UX by default
3. Vector retrieval scoring — same formula `1/(1+d)`
4. Graph guidance quality thresholds — same values
5. Keyword boost — same logic and amounts
6. Answer generation prompt — same system prompt
7. Evidence image loading — same strategy

## 17. Safety Requirements

### Must NOT

- Change vector retrieval scoring algorithm
- Change graph guidance quality thresholds
- Change keyword boost values
- Modify existing LanceDB or Neptune data
- Change the answer generation system prompt
- Break existing CLI commands
- Remove any existing output format
- Modify parser code
- Modify graph extraction code
- Modify existing experiment outputs

### Must

- Default mode produces identical output (no visible change without flags)
- All new features are opt-in (flag-activated)
- Graph confidence filtering is conservative (threshold 0.0 = pass everything)
- Isolation violations are warnings by default, errors only with `--strict`
- Trace data never changes the answer (observation only, no side effects)
